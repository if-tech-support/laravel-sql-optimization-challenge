# 計測のやり方（共通）

最適化は「**速くなったかを必ず数字で確認する**」のが大原則です。
このドキュメントでは、各課題で使う 3 つの計測手段を説明します。

> 体感ではなく「クエリ本数」「実行時間」「EXPLAIN」で語れるようになることが、この課題のゴールです。

---

## 1. Debugbar（ブラウザで見る）― まずはこれに慣れる

このリポジトリには [barryvdh/laravel-debugbar](https://github.com/barryvdh/laravel-debugbar) が入っています。
`APP_DEBUG=true`（`.env` の初期値）であれば、課題ページを開いたとき **画面のいちばん下に黒っぽい横長のバー**が出ます。これが Debugbar です。

> バーが見当たらないとき: 画面右下にある小さなアイコン（Laravel ロゴ）をクリックすると開きます。
> それでも出ない場合は `.env` の `APP_DEBUG=true` とブラウザの再読み込みを確認してください。

### バーの構造（どこに何が表示されるか）

バーは「左側のタブ」と「右側の数字」に分かれています。実際の表示はおおよそ次のようになっています。

```
┌─ 画面下部のバー ───────────────────────────────────────────────────────────────┐
│ Request │ Timeline │ Views (2) │ Queries (3) │ Models (1000)     13.x  4MB  123ms  GET /challenges/01-select │
└──────────────────────────────────────────────────────────────────────────────┘
   └──── タブ（クリックで中身が開く）────┘                  └─── 右側の数字（クリック不要、常に見える）───┘
```

**右側の数字**（バーを開かなくても常に見える、まず最初に見るべき指標）

| 表示例 | 意味 |
|--------|------|
| `4MB`   | このページ生成で使った **メモリのピーク**。SELECT * や N+1 で膨らみます。 |
| `123ms` | このページの **生成にかかった時間**（ミリ秒）。小さいほど速い。 |
| `GET /challenges/01-select` | 今表示しているリクエスト。 |

**左側のタブ**（クリックすると下に詳細が開く）

| タブ | クリックすると見えるもの | 注目する課題 |
|------|------|------|
| **Queries (3)** | 発行された SQL の一覧。**カッコ内の数字が「クエリ本数」**。N+1 だとここが何百本にもなり、同じ形の SQL が延々と並びます。 | 02 / 03 / 04 |
| **Models (1000)** | DB から **メモリに展開された Eloquent モデルの件数**。全件取得などで膨らみます。 | 01 |
| **Timeline** | 処理ごとの所要時間の内訳。 | 06 |
| **Request** | URL・コントローラ・ステータスコードなどの基本情報。 | （参考） |

### 基本の操作手順（毎回これを繰り返す）

1. 課題ページをブラウザで開く。
2. 画面下部のバーで、**右側の数字**（`○ms` `○MB`）と、必要なら **タブ**（`Queries` など）をクリックして数字を控える。← これが「最適化前（Before）」。
3. `docs/0X-*.md` の指示どおりコードを直す。
4. **ブラウザを再読み込み**（Mac は `⌘ + R`、Windows は `Ctrl + R` または `F5`）する。コードを直しただけでは画面の数字は変わりません。**必ず再読み込み**してください。
5. 数字がどう変わったか（After）を Before と比べる。

> どの数字に注目するかは課題ごとに違います。
> 課題02 / 03 / 04（N+1・集計）は **`Queries` タブのクエリ本数**、
> 課題01 は **`○MB`（メモリ）と `Models` タブの件数**、
> 課題06 は **`○ms`（実行時間）** を主に見ます。
> 課題05 はバーではなく**ページ内に表示される EXPLAIN の表**を見ます（後述）。

## 2. クエリログ（コードで取る）

`tinker` やコントローラ内で、発行クエリを配列として取得できます。

```bash
./vendor/bin/sail artisan tinker
```

```php
use Illuminate\Support\Facades\DB;

DB::enableQueryLog();

// 計測したい処理
App\Models\Order::with('user')->limit(100)->get();

$log = DB::getQueryLog();
count($log);   // 発行されたクエリ本数
$log[0];       // 1 本目の SQL とバインド値
```

実行時間をざっくり測るなら：

```php
$start = microtime(true);
// 計測したい処理
$ms = (microtime(true) - $start) * 1000;
printf("%.1f ms\n", $ms);
```

## 3. EXPLAIN（インデックスの効きを見る）

クエリがインデックスを使えているか、フルスキャンになっていないかは `EXPLAIN` で確認します。
課題05 で必須です。

ビルダの生 SQL を取り出して MySQL で EXPLAIN します。

```php
$query = App\Models\Order::where('user_id', 1234)->where('status', 'paid');

DB::select('EXPLAIN ' . $query->toSql(), $query->getBindings());
```

または MySQL に直接入って実行します。

```bash
./vendor/bin/sail mysql
```

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 1234 AND status = 'paid' ORDER BY ordered_at DESC;
```

### EXPLAIN の読みどころ

| 列 | 注目ポイント |
|----|------|
| `type` | `ALL` = フルスキャン（遅い）。`ref` / `range` / `const` に近いほど良い。 |
| `key` | 実際に使われたインデックス。`NULL` ならインデックス未使用。 |
| `rows` | 走査見込み行数。小さいほど良い。 |
| `Extra` | `Using filesort` / `Using temporary` が出たら並び替え・集計の改善余地あり。 |

> インデックスを張る前後で `type` と `rows` がどう変わるかを、必ずスクショかメモで残しておきましょう。PR の説明に貼ると説得力が出ます。
